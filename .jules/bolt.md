## 2024-03-24 - [Sidebar Search Main Thread Block]
**Learning:** Even with small client-side datasets, executing search filtering and DOM manipulation on every input keystroke (`input` event) can block the main thread and impact TTI, especially on lower-end devices. The DOM creation and appending operations inside the event listener compound the issue.
**Action:** Always wrap frequent user input events (like search or scroll) that trigger DOM updates or array filtering in a debounce or throttle function. In this case, adding a simple 300ms debounce timeout to the search input dramatically reduces unnecessary execution.
## 2026-04-14 - [Redundant toLowerCase() calls in search filter loop]
**Learning:** Calling `toLowerCase()` inside a loop (especially a filter loop that executes frequently, such as on every keystroke/debounced keystroke) creates redundant string allocations and slows down execution. In our benchmark with 1 million iterations, pre-computing the lowercased strings reduced execution time from ~770ms to ~436ms, yielding a ~43% improvement.
**Action:** Always pre-compute derived values like lowercased strings outside of fast-path loops if the source data is static or updated infrequently.

**Learning:** In heavily used functions like translation resolvers (`useTranslations`), repeating array allocation via `split('.')` significantly degrades performance. Caching the array splits in a module-level `Record` (when the memory footprint of unique keys is low) yields a substantial performance improvement (e.g., from ~1050ms to ~734ms for 1,000,000 ops). Furthermore, bypassing `.split()` for keys that don't contain a dot via a simple `indexOf('.') === -1` check reduces overhead.
**Action:** When working with repeated string transformations (like dot-separated deep-property accessors), consider introducing a lightweight local cache (`Record` or `Map`) for intermediate results, rather than relying on native optimization of `.split()`.
## 2024-04-18 - [Deferring Heavy Background Animations]
**Learning:** [Heavy Javascript files like Vanta.js and Three.js drastically impact the Total Blocking Time and critical render path. Standard `<script defer>` tags might still be parsed too early by Lighthouse, resulting in poor TBT, FCP, and LCP scores.]
**Action:** [Instead of using standard defer tags, use `requestIdleCallback` coupled with `setTimeout` to dynamically inject script elements to the DOM after the initial page load (`window.onload`) is complete to achieve flawless performance metrics without removing visual components.]

## 2024-04-18 - [Splitting Monolithic CSS files]
**Learning:** [Splitting a massive, monolithic global CSS file (like global.css) into component-specific styles without a full architectural redesign leads to massive visual regression and syntax errors due to tightly coupled styling rules, selectors, and variables.]
**Action:** [Unless a complete CSS architectural rewrite is in scope, do not attempt to lazily slice a monolithic CSS file, as the performance gains (unused CSS savings) are negligible compared to the visual breakages.]
